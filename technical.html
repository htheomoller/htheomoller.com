Murmur: Architecture Skeleton

How Murmur is built, and why each piece exists.

Abstract

Murmur is a WhatsApp-based collective memory system built for small teams. One WhatsApp number, many people talking one-to-one, a shared knowledge base that anyone can query. This document describes the technical architecture: the message pipeline, multi-tenant design, memory engine, personality system, and the economics of a bring-your-own-key model.

Five foundational decisions shaped the architecture. Build a custom core rather than forking an existing framework. Connect directly to Meta’s Cloud API rather than through a third-party broker. Store all memory in a database with vector search rather than in files. Design for multi-tenancy from day one. Enforce role-based access control with private memory at the data layer.

Every design choice here was tested first in a single-tenant deployment, a WhatsApp assistant running for a wedding venue in southern Brazil. Eight months of production incidents shaped this blueprint.

01

The pipeline

Every message follows the same path through Murmur. A WhatsApp message arrives via webhook. The system checks for duplicates at the database level because Meta retries webhooks aggressively, and processing a message twice means a user gets two responses. After dedup, the system resolves which tenant owns the phone number, identifies the sender, checks their role and permissions, then assembles context. 

Incoming WhatsApp Message
  → Webhook Handler (dedup, validate)
  → Tenant Resolver (phone number → tenant)
  → Identity Resolver (sender phone → user + role)
  → RBAC Gate (owner/member/viewer permissions)
  → Context Builder (personality + memories + history)
  → Agent Runtime (LLM call + tool execution)
  → Memory Evaluator (is this worth remembering?)
  → Knowledge Capture (extract → embed → store)
  → Response Formatter (split, batch, send)
  → Outgoing WhatsApp Message

Context assembly is where the system earns its keep. The context builder loads four personality files that define how the assistant thinks and speaks, retrieves the twenty most semantically relevant memories from the knowledge base, and includes recent conversation history. Total prompt size sits around twelve to sixteen thousand tokens.

The language model generates a response using the tenant’s own API key (more on that in section seven). Before the response is sent, the system writes a crash-safe extraction record. If the process dies between sending the response and extracting memories, nothing is lost. A background worker picks up pending extractions and processes them, classifying knowledge into categories, generating embeddings, checking for contradictions against existing memories, and storing what’s worth keeping.

Full round-trip: two to four seconds. The language model call dominates.

03

Multi-tenant design

A tenant is one Murmur instance. One WhatsApp number, one shared brain, one set of personality files, one pool of memories. Each tenant has N users with one of three roles: owner, member, or viewer.

Isolation at the database level

Every table has a tenant_id column. Every query filters by it. Row-level security policies enforce the boundaries in Postgres itself, so even a bug in the application layer cannot leak data across tenants.

Tenant configuration is cached in memory and refreshed every sixty seconds. Personality files are cached per tenant and invalidated through Supabase Realtime subscriptions when an owner edits them. The webhook handler resolves an incoming phone number to a tenant in microseconds.

Permission matrix

Action Owner Member Viewer
Send messages ✓ ✓ ✓
Query the knowledge base ✓ ✓ ✓
Conversations remembered ✓ ✓
Can teach (create memories) ✓ ✓
Private memories ✓ ✓
Invite members ✓
Manage personality ✓

When one VPS maxes out, spin up a second and route tenants by region. No architectural changes needed.

04

The memory engine

Crash-safe extraction

Memory extraction is the most failure-prone path in the system. If the process crashes between a conversation turn and memory extraction, the user said something worth remembering and the system silently forgot it. This happened in production with the single-tenant deployment.

The fix is a crash-safe queue. Before sending a response, the system writes the message-response pair to a pending extractions table. A background worker picks up pending records, runs a language model evaluation to extract knowledge, and stores whatever is worth keeping. If the worker crashes, it retries on restart. After three failed attempts, the record is flagged for manual review.

Memory categories

Category What goes here Example
Fact Verifiable information “Project deadline is March 15”
Decision Choices made “We’re going with React for the frontend”
Preference How someone likes things “Client prefers morning meetings”
Commitment Promises or tasks “João will send the proposal by Friday”
Status Current state of something “Server migration is 60% complete”
Deadline Time-bound items “Tax filing due April 30”

Each memory carries metadata: who said it, when, a confidence score, topic tags, and a visibility flag. Visibility is either shared or private. The language model detects privacy signals in conversation (“keep this between us,” “don’t tell anyone”) and marks those memories accordingly.

Semantic context loading

When building context for a conversation, the system generates an embedding from the user’s latest message and retrieves the twenty most relevant memories using HNSW vector search. HNSW was chosen over IVFFlat because it handles incremental inserts (new tenants start with empty memory) and delivers roughly fifteen times faster queries. IVFFlat only justifies itself at fifty million vectors.

Private memories are enforced at the SQL level. The RLS policy filters them so a user only ever sees shared memories plus their own private memories. The language model never even receives another user’s private memories in its context window.

05

Personality system

Each assistant’s personality is defined by six markdown files, ordered by cache stability.

File Purpose Tokens Cache priority
IDENTITY Name, role, language, key business facts ~500 High (rarely changes)
SOUL Values, personality, tensions, humor ~1,500 High
VOICE Tone, vocabulary, sentence patterns, always/never rules ~1,500 High
AGENTS Operational rules: batching, escalation, dormancy ~1,000 High
EXAMPLES Reference conversations showing ideal behavior ~2,000 Medium (conditional)
USER Business context, key people, products ~1,500 Medium (conditional) 
[SYSTEM INSTRUCTIONS]           ← Framework (never changes)
[IDENTITY.md]                   ← Name, role (rarely changes)
[SOUL.md]                       ← Personality (rarely changes)
[VOICE.md]                      ← Tone rules (rarely changes)
[AGENTS.md]                     ← Operational rules
─── cache boundary (above = cached 85%+) ───
[Relevant memories]             ← Changes per conversation
[Conversation history]          ← Changes per message
[USER.md / EXAMPLES.md]         ← Loaded conditionally
[Runtime metadata]              ← Timestamp, user info, role

The first four files are always loaded with the most stable content first. This maximizes prompt caching within a single tenant’s conversation stream. The cache prefix stays identical call after call until the personality changes, which for most tenants means months.

New tenants get a default personality: neutral-friendly, professional but warm. Personality can be customized through a Soul Interview, a ten-minute guided conversation via WhatsApp. The assistant asks what kind of personality would fit, offers archetypes, calibrates formality and tone, and generates the first three files automatically.

06

A message trace

What happens when a team member named João sends “The client meeting is moved to Thursday 3pm.”

The webhook arrives from Meta. The system checks the dedup table. This message ID hasn’t been seen, so it proceeds. The phone number resolves to the Acme Project tenant. João is identified by his phone number and his role comes back as member, meaning he can teach the system and his conversations are remembered.

The context builder loads Acme’s personality files from cache, generates an embedding for João’s message, and retrieves twelve relevant shared memories plus three of João’s private notes. It loads the last twenty messages from this conversation. Total context: about nine and a half thousand tokens. Eighty-five percent were cached from João’s previous message.

The language model responds: “Got it! I’ve updated the client meeting to Thursday at 3pm. Should I remind everyone the day before?”

Metric Value
Input tokens 9,800
Output tokens 42
Cache read 8,300 (85%)
Latency ~3 seconds

Before sending, the system writes the message pair to the pending extractions queue. The response goes out. The background worker picks up the extraction, identifies a fact worth remembering, generates an embedding, checks existing memories, finds a conflicting one (“client meeting is Tuesday 2pm”), marks it as superseded, and stores the new one.

Three seconds. João never noticed any of it.

07

BYOK architecture

BYOK, Bring Your Own Key, means users provide their own language model API key. Murmur handles routing, memory, and infrastructure. Users pay their own inference costs directly to Anthropic or OpenAI.

API key security follows the same pattern as WhatsApp credentials. Encrypted at rest with pgcrypto. Never logged, never included in error messages. Decrypted only at the moment of the language model call, held in memory briefly, then discarded. Invalid or expired keys trigger a friendly WhatsApp message pointing the user to the dashboard.

08

Security and privacy

Two risks sit above everything else: tenant data cross-contamination and private memory leakage. Both are mitigated at the database level, not the application level. RLS policies on every table. Tenant ID in every query. The application layer is a second defense, not the first.

Threat model

Threat Mitigation Severity
Tenant data cross-contamination RLS policies on all tables. Tenant ID in every query. Critical
Private memory leakage Visibility filter at SQL level. LLM never sees others’ private memories. Critical
Prompt injection No code execution. Tools limited to DB queries. System prompt hardened. High
Credential theft Encrypted at rest (pgcrypto). Never logged. High
Webhook spoofing Verify Meta webhook signature on every request. High
DDoS via webhook flood Rate limiting per phone number. Async queue with backpressure. Medium

Data protection (LGPD/GDPR)

Deletion cascades through all user memories. Export generates a markdown file with shared and private memories. The first message triggers an explicit consent flow. Only extracted memories are stored, not full conversation logs. Audio from voice notes is deleted immediately after transcription. The system never retains recordings.

An audit log tracks every significant action: memory creation and deletion, user invitations, role changes, data exports. Breach notification follows the seventy-two hour GDPR requirement with structured alerting.

09

Graceful degradation

Every external dependency has a failure mode and a user-facing response.

Failure User experience System behavior
LLM API down “I’m having trouble thinking right now.” Queue + retry every 60s. Alert owner after 5 min.
API key invalid “Check your Murmur dashboard.” Log error. Notify tenant owner.
Supabase unreachable Silence (~5s) Webhook returns 500 → Meta retries. Systemd auto-restart.
Whisper timeout “Could you type that out?” Log error. Conversation continues.
Embedding fails Normal conversation Memory saved without vector. Background retry.
Process crash Silence (~5s), then resumes Systemd auto-restart. Pending extractions retried.

A health endpoint returns active tenant count, uptime, pending extraction queue depth, and the last error. Monitored externally at five-minute intervals.

10

Build order

The architecture is built in phases. This matters because Murmur is being built by a non-developer using AI tools, so the timeline must be realistic.

Phase 0: Foundation (weeks 1–3)

Database schema, webhook server, message dedup, basic WhatsApp send and receive, BYOK key storage, health check endpoint.

Milestone: a webhook receives a message, logs it, and sends a response.

Phase 1: Single-tenant MVP (weeks 4–8)

Context builder, agent loop, crash-safe memory extraction, semantic search, token tracking, conversation consolidation, voice notes, graceful degradation.

Milestone: one working assistant with memory, running on one number for one team. This is the product validation point. Get one real team using it before building anything else.

Phase 2: Multi-tenant (weeks 9–14)

Tenant resolution, config caching, per-tenant personality and memory isolation, user identification, RBAC enforcement, private memory, invitation flow.

Milestone: two tenants, three roles, private memory, one process.

Phase 3: Onboarding (weeks 15–20)

Soul Interview, Embedded Signup, web signup flow, admin dashboard.

Milestone: a customer can sign up, pay, connect WhatsApp, and have a working assistant.

Phase 4: Proactive intelligence (weeks 21–25)

Cron scheduler, daily digests, deadline reminders, WhatsApp template messages.

Milestone: the assistant reaches out on its own.

Phase 5: Hardening and launch (weeks 26–30)

Security audit, LGPD/GDPR compliance, legal documents, load testing, monitoring, first five pilot teams, public launch.

The Meta Tech Provider application runs in parallel from day one. It is the longest lead-time item and entirely outside anyone’s control.
