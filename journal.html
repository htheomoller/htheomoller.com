Building a Memory System

We designed a brain, critiqued it, then built the damn thing. Here’s what happened on day one.

The Critique

Murmur needs memory. Not the kind where you dump everything into a vector database and hope cosine similarity does the rest — actual memory, modeled on how brains work. A hippocampus for fast capture, a neocortex for slow consolidation, and a retrieval function that gets better with use.

We designed the schema: a memories table backed by pgvector, an activate() function that scores results using ACT-R decay from cognitive psychology, Hebbian reinforcement that strengthens memories each time they’re recalled, and confidence weighting. The math was sound. The SQL was clean.

Then we tore it apart. A formal critique identified seven gaps between what we’d designed and what would actually work.

# Gap The Problem
1 No write path A beautiful retrieval function with zero memories to retrieve. No pipeline to embed content and store it.
2 Vector-only search Pure semantic search drops ~22% precision vs. hybrid. Searching for “Chakra decision” needs keyword matching, not just vibes.
3 No multi-tenant columns Building single-tenant and hoping to bolt on multi-tenancy later — one of the oldest mistakes in SaaS.
4 No lifecycle management Wrong memories persist at confidence 1.0 forever. No way to mark something as superseded.
5 No consolidation engine The neocortex — the whole differentiator — was a wish, not a design.
6 Dead importance column Nullable, unused in scoring, no defined semantics. Technical debt from day zero.
7 No retrieval logging No way to know if activate() was returning useful results or garbage.

Harsh, but honest. And that’s the point of critiquing your own work before anyone else gets to — you find the structural problems while they’re still cheap to fix.

What We Built

After the critique, we went back and addressed each gap. Not all of them — some were deliberately deferred — but most.

Prescription Status What We Did
Write path Done Built embed-and-store, a Supabase Edge Function that accepts markdown content, calls OpenAI’s embedding API, and inserts into the memories table. Every memory capture now dual-writes: vault file first, then HTTP POST to Supabase. Fire-and-forget — the vault file is always the source of truth.
Hybrid search Done Rewrote activate() to combine vector similarity with BM25 keyword matching using Reciprocal Rank Fusion. The scoring formula became: rrf_score × softplus(ACT-R_decay) × confidence × importance. Both signals contribute.
Multi-tenant columns Deferred The dogfood is single-user. Adding tenant columns now would be architectural theater — correct in principle, zero value today. We’ll add them when Murmur goes multi-tenant.
Lifecycle management Done Added status column with three states: active, superseded, archived. Added superseded_by reference. activate() now filters to active memories only. Old knowledge can be corrected without being deleted.
Consolidation engine Not yet Still conceptual. The neocortex — canonical_views table, consolidation logic, sleep replay — is the next major design challenge. We need more data before we can design it well.
Importance column Done Integrated into the scoring formula. Defaults to 1.0, adjustable per memory. No longer dead weight.
Retrieval logging Done Created activation_log table. Every call to activate() logs the query, result IDs, and scores. This is the ground truth for measuring whether the system actually works.

Five out of seven addressed. The two we deferred — multi-tenant columns and the consolidation engine — are both cases where building prematurely would be worse than waiting. Multi-tenancy adds complexity with no current user. Consolidation needs a corpus large enough to consolidate.

With the infrastructure in place, we set up an A/B test: run both retrieval systems side by side for two weeks and see which one finds what the other misses.

A/B Test — Day 1

The test pits two retrieval systems against each other on the same corpus of memories:

QMD (control) activate() (experimental)
Engine BM25 + vector + cross-encoder reranking RRF (vector + BM25) + ACT-R decay + confidence
Embedding QMD’s built-in model OpenAI text-embedding-3-small (1536d)
Reinforcement None Hebbian (access_count + last_accessed_at)
Lifecycle None (all notes equal) status filter (active/superseded/archived)

Infrastructure: healthy

76 memories in Supabase, up from 69 seeded — 7 new via dual-write today. The dual-write pipeline is working: new memories hit both the vault and Supabase. Both endpoints responding correctly: search-memories for retrieval, embed-and-store for capture.

Test data: thin

Only 3 entries in the activation log so far:

Query Top Score Results
“baileys bridge deprecated” 0.080 3
“dual write test verification” 0.114 3
“A/B test activation log memory retrieval” 0.036 5

Three data points. Not much to draw conclusions from. Day 1 was mostly other work — not heavy memory-retrieval sessions.

Observations

Live comparison

We ran both systems on the same query. Here’s what happened:

QMD found the actual A/B test protocol, the implementation plan, CLAUDE.md, PROJECT-MEMORY.md — the documents about the test. Strong on project-level search.

activate() found related memories — a Lia bugs incident, a Murmur architecture decision, a token optimization decision. Relevant context, but nothing directly about the A/B test itself.

Why the difference? The Supabase brain only has Memory/ folder content (76 files). QMD indexes the entire vault — hundreds of files across projects, research, and personal notes. For project docs and research, QMD wins by default because Supabase simply doesn’t have that content.

Three things worth watching

Scores are low. All under 0.12. Could be normal for ACT-R decay on fresh memories — low access counts mean low activation — but worth tracking as the corpus grows.

The logging gap. The activation_log table only tracks activate() calls. It doesn’t log the QMD side, so there’s no built-in side-by-side comparison. We’re relying on noting differences inline during sessions.

Different scopes. QMD searches the whole vault. Supabase only has memories. They’re not competing on equal ground. Supabase should shine for memory-specific retrieval — corrections, decisions, facts — while QMD handles everything else.

Verdict

Too early to judge performance. Three data points isn’t enough. The plumbing works, the dual-write is live, but we need more retrieval-heavy sessions to get meaningful signal. The real test will come when we’re deep in a project and need to recall past decisions or corrections quickly.

13 more days until the checkpoint (March 23). We’ll get there.
